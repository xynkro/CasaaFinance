# FinancePWA / CasaaFinance — Architecture Audit
> **Date:** 2026-06-06 · **Method:** 30-agent council workflow (`.state/audit_workflow.mjs`) · **Funnel:** 45 raw → 17 triaged → **5 confirmed** (adversarially verified at 1–2 user scale).
---
## Executive summary
This audit reverse-engineered the entire system with a 30-agent council — 6 subsystem mappers, then 5 review lenses (performance, architecture, scalability, maintainability, security), a triage/dedupe pass, and an **adversarial verification pass tuned to KILL false-positives and millions-grade over-engineering**. This is a 1–2 user personal tool, so the bar for every finding was *value to a solo maintainer*, not enterprise polish.

**Funnel: 45 raw findings → 17 deduped → 5 confirmed.** 40 were dropped — most as over-engineering that doesn't belong in a personal app (Redis, queues, a DB, microservices). What survived is what's actually worth your time.

**The headline is one CRITICAL finding: your and Sarah's entire portfolio is world-readable right now** — holdings, cost basis, account sizes, and P&L — by anyone with the deployed PWA URL. Everything else is medium/low and about runtime robustness and maintainability.
## Northstar
Make Caspar + Sarah serious money in the market through **real-time, timely, actionable, accurate, specific intel** across **ETFs, stocks & options**.

*(Note: the audit agents auto-synthesized a process-focused northstar from the code — "produce a disciplined daily plan / never trade on its own." Caspar corrected this: that's a guardrail, not the goal. The goal is **making money** via good intel.)*

## Operating principles (reverse-engineered)
1. Google Sheets (~50 tabs, one hardcoded SHEET_ID) is the single source of truth and the only inter-process bus — Python WRITES via gspread/OAuth, the PWA READS via the public gviz CSV endpoint with no backend server and no API key on the read path (pwa/src/data.ts:3-7, src/sheets.py).
2. Headless cron does the work; the PWA only reads. 31 scheduled GitHub Actions workflows + 2 Mac LaunchAgents populate tabs; the PWA never writes back to Sheets — user decision actions (filled/killed/deferred) are localStorage-only and the documented v2 Sheets write-back never shipped (pwa/src/lib/decisionActions.ts:7-20).
3. Paper-trading only — the human executes real trades. The single closed automation loop (build_daily_plan -> daily_plan tab -> alpaca_paper_execute) has a HARD PAPER GUARD that refuses any non-paper Alpaca URL and exits 2 (scripts/alpaca_paper_execute.py:262-267); IBKR is read-only position discovery, not order placement.
4. Motley Fool / macro / news / insider / gov-trades are ENGINE INPUTS, never auto-signals. MF picks are ingested out-of-band (Claude+Chrome MCP) into curated_picks and only TILT the plan; macro surprises write a lean to macro_lean that build_daily_plan uses as a sizing tilt — none of them auto-place orders (scripts/ingest_curated_picks.py, scripts/build_daily_plan.py).
5. Shared scoring core, parallel scan engines: indicators.compute_indicators + technical_score.compute_scores (16 signals, per-strategy weights) are the one clean choke-point every engine imports — but TWO CSP/CC scanners (daily_options_scan vs daily_tracker->option_scanner) run with DIFFERENT delta targets and DIFFERENT IV sources and both write scan_results, the single biggest maintainability hazard (src/option_scanner.py:43-46 vs scripts/daily_options_scan.py:42-45).
6. Free, keyless-where-possible data sources, absorbed by retry not rate-limiting: yfinance / FinViz / TradingView / CBOE / Finnhub / Alpaca-paper, with NO caching layer anywhere — the only defense against same-minute writer storms and 429s is a monkey-patched gspread backoff (src/sheets.py:44-74). For 1-2 users this is fine; do NOT add Redis/queues/a DB — that would be millions-scale over-engineering.
7. Append + SGT audit-timestamp + client-side 'latest': writers stamp every date PK with an SGT HHMMSS suffix so Mac-local and GH-UTC writes sort chronologically (src/schema.py:46-48), tabs grow unbounded, and the PWA re-derives 'latest row per date' in the browser (data.ts latest/dedup). A one-shot cleanup_dupes.py janitor is the manual GC.

## End-to-end flow
- 1. SCHEDULE: GitHub Actions cron (all UTC) fires 31 scheduled workflows; 2 surviving Mac LaunchAgents (com.caspar.ibkr-grab 05:30 SGT needing TWS, com.caspar.daily-tracker 06:00 SGT) own IBKR new-position discovery the cloud can't see. Each workflow runs checkout -> setup-python 3.12 -> pip install -> python scripts/<x>.py.
- 2. AUTH: every script calls src.sync.load_env() then src.sheets.authenticate(), which resolves one OAUTH_TOKEN_JSON secret (CI) into a gspread client and monkey-patches HTTPClient.request with 429/5xx backoff — the single auth chokepoint and single blast-radius point for all 31 workflows.
- 3. PULL: scanners fetch free market data — yfinance (prices, OHLCV history, option chains, .info, .calendar, ^VIX/^GSPC), FinViz (high-IV + momentum discovery), TradingView scanner (tv_signals), CBOE (gex_regime), Finnhub (news/calendars/analyst), USAspending+CapitolTrades (gov/congress), Alpaca-paper (benchmark) — no key on most paths, no cache, ~5-8 yfinance round-trips per ticker.
- 4. SCORE: src/indicators.compute_indicators(df) derives ~30 technical fields from one OHLCV frame; src/technical_score.compute_scores() maps them to 16 normalized signals and applies per-strategy weights to a [-100,+100] score -> 0-100 conviction. Both scan engines import this shared core.
- 5. SCAN: daily_options_scan.py (02:35 UTC) runs a macro gate (STANDARD/CAUTION/HALTED) then scans the watchlist across ALL strategies (CSP/CC/LONG_CALL/IC/PCS/CCS/PMCC) plus FinViz high-IV discovery (HARVEST_CSP), solving its own BSM delta/IV; daily_tracker.py builds the portfolio snapshot then invokes the SECOND scanner src/option_scanner.scan_watchlist (different deltas, Yahoo IV). Both append S.ScanResultRow to scan_results/harvest_scan plus snapshots/positions/options/technical_scores/exit_plans.
- 6. BRAIN (LLM inputs, not auto-signals): 4 'brain' workflows (daily-brief, wsr-lite, wsr-full) invoke anthropics/claude-code-action (Opus, pinned, $0 on Max) driven by prompts/cron_*.md to synthesize briefs/WSRs into tabs + Drive PDF archive; Motley Fool picks are read out-of-band via Chrome MCP and written to curated_picks by ingest_curated_picks.py.
- 7. MACRO/NEWS LANE: trigger_alerts pulls Finnhub news + economic calendar, merges RSS (WSJ/FT/Bloomberg/MarketWatch/CNBC/NYT/SeekingAlpha) + newsdata.io + Bloomberg-via-Gmail (gmail_client, read-only), flags hot headlines, applies a ±15min high-impact blackout window, and writes net macro lean to the macro_lean tab.
- 8. AGGREGATE PLAN: build_daily_plan.py (14:00 UTC) reads scan_results + screen_candidates + curated_picks + snapshot_caspar (NLV) + macro_lean, prepends a hardcoded STANDING_ALLOCATION (QQQ/VIXM/IEF/GLD), tilts by macro lean + MF picks, ranks by conviction, and writes the daily_plan tab (clear + rewrite today).
- 9. PAPER EXECUTE: alpaca_paper_execute.py (14:30 UTC) reads daily_plan, enforces the HARD PAPER GUARD (refuse non-paper URL, exit 2), places deterministic-client-order-id orders on Alpaca PAPER, and writes fill outcomes back onto daily_plan — the one closed loop, fully fenced to paper.
- 10. ALERT: trigger_alerts.py (*/10 13-21 UTC, M-F) re-evaluates each WATCHING decision (gate state-machine mirrored from the PWA's evaluateTrigger), dedups via the trigger_alerts state tab (decision_key = date|account|ticker|strategy|strike), and on an act_now transition fires Telegram via telegram.send() into one of 4 forum topics (Multi Day Swing=3, Macro News=6, Options Intel=492, Insider=510); scanners and signal_feedback also push.
- 11. READ: the GitHub-Pages PWA (deployed by deploy.yml on push to pwa/**) gates on a localStorage PIN, then App.tsx calls data.ts fetchDashboard() on mount + every 15 min + on every tab-foreground, firing ~39 gviz CSV fetches in two sequential Promise.all batches (each .catch(()=>[]) for graceful degrade), pulling FULL tab history (~9.5MB/load) and reducing to 'latest' client-side into one DashboardData object.
- 12. RENDER + HUMAN EXECUTES: the single DashboardData object is prop-drilled into ~7 pages / ~41 cards (no router, no store); the user sees Today's Plan, decisions, portfolio P&L, scans, and brief/WSR, records fills/kills/defers to localStorage only (lost on cache clear, invisible to the brain), and places any REAL trades by hand in IBKR/their broker.

## Subsystem maps

### Backend data / Google Sheets layer — a single Google Sheet (~50 tabs) is the system's database. Python scripts (cron on GitHub Actions + Mac-tethered jobs) WRITE to it via gspread/OAuth; the React PWA READS it directly via the public gviz CSV endpoint (no backend server, no API key on the read path). schema.py defines all tab schemas as 48 dataclasses + 12 input→row mappers. There is NO read-back helper in Python's sheets.py — every consumer script reads the whole tab with get_all_values().
**Data flow:**
- WRITE (Python→Sheet): A producer (IBKR grab, Finnhub calendars, TV scanner, brain WSR output, scanners) builds typed dataclass rows from a JSON ledger/sidecar via schema.py mappers (e.g. snapshot_caspar_from_grab, decisions_from_ledger at src/schema.py:1548-1880).
- Each dataclass.to_row() serializes to a list[str], formatting numbers via _num() and stamping the date column with an SGT audit suffix (_ts_suffix → 'YYYY-MM-DDTHHMMSS', src/schema.py:46-48). SGT anchoring (src/schema.py:18-23) is deliberate so Mac-local and GH-Actions-UTC writes sort into one chronological order lexicographically.
- sheets.authenticate() (src/sheets.py:77) returns a gspread client via 3 credential paths: OAUTH_TOKEN_JSON env (CI), GOOGLE_SERVICE_ACCOUNT_JSON env, or local cached OAuth token. It also monkey-patches gspread's HTTPClient.request with exponential-backoff retry on 429/5xx (src/sheets.py:44-74).
- APPEND-style tabs (snapshots, positions, options, macro): sync.py calls sh.ensure_headers() then sh.append_row()/append_rows() (src/sheets.py:134-173). ensure_headers is called before EVERY append — each call does an open_by_key + worksheet + row_values(1) round-trip.
- UPSERT-style tabs (decisions, earnings, economic, analyst, news, insider, api_usage, trigger_alerts, live_prices, gov_confluence...): each script reads the ENTIRE tab with ws.get_all_values(), filters out rows whose key matches the new batch in Python, ws.clear()s the tab, then ws.update('A1', kept+new) to rewrite it whole (e.g. scripts/push_decisions.py:217-242, scripts/finnhub_calendars.py:210-231).
- READ (Sheet→PWA): pwa/src/data.ts builds a gviz CSV URL per tab (csvUrl, line 5-6) against a HARDCODED SHEET_ID (line 3) and a hand-maintained GID lookup table (GIDS, lines 14-76). fetchTab() does fetch() + Papa.parse with header:true (lines 78-93).
- fetchDashboard() (data.ts:1379) fires ~39 separate gviz fetches in two Promise.all batches, each wrapped in .catch(()=>[]) so any one tab failing degrades gracefully. Returns a single giant DashboardData object (~70 fields).
- Client-side shaping: latest()/latestGroup()/dedup()/sortByDate() (data.ts:1354-1377) reduce full tab history to 'latest row(s) per date' in the browser. summarizeNews/summarizeInsider/lookupTvConsensusMap/indexLivePrices build per-ticker Maps. There is NO server-side filtering — the full history of every tab crosses the wire each load.
- REFRESH: App.tsx re-runs the full 39-fetch load every 15 min via setInterval AND on every tab-foreground (visibilitychange), with no staleTime/dedup cache — no React Query, just raw fetch (App.tsx:48-58).
- WRITE-BACK GAP: the PWA has read-only Sheets access. User decision actions (filled/killed/deferred) are journaled to localStorage only (pwa/src/lib/decisionActions.ts) and joined client-side at render. The documented v2 'sync to Sheets via Apps Script/Cloudflare Worker' (decisionActions.ts:18-21) never shipped — so action state is device-local and lost on cache clear. The only PWA→cloud write is a GitHub workflow_dispatch POST to re-trigger crons (SettingsPage.tsx:658-670).

**Smells:**
- 39 separate gviz CSV fetches per dashboard load, every 15 min + on every tab-foreground, with zero caching. This is the single biggest latency/rate-limit exposure: a foregrounding burst fires 39 concurrent requests to docs.google.com. gviz is undocumented/unthrottled-by-Google and can soft-fail under burst, and each fetch pulls the ENTIRE tab history (not just latest). For a 2-user app this mostly 'works', but it's the most likely thing to flake. — `pwa/src/data.ts:1379-1453 (two Promise.all batches, 39 fetchTab/fetchTabByName calls counted via grep); App.tsx:51-58 (setInterval 15*60*1000 + visibilitychange both call load() with no staleTime/dedup).`
- UPSERT = read-whole-tab + ws.clear() + rewrite-whole-tab, duplicated ~12 times across scripts with near-identical bodies. Two problems: (1) the clear-then-update is non-atomic — a crash between ws.clear() and ws.update() truncates the tab to empty; (2) it's O(tab) per write and the logic is copy-pasted instead of living in sheets.py. A single shared upsert(client, tab, rows, key_fn) helper would remove ~12 copies. — `scripts/push_decisions.py:241-242 (ws.clear() then ws.update('A1', keep_rows)); scripts/finnhub_calendars.py:230-231 (identical clear()+update); 12 upsert defs found via grep across finnhub_calendars/api_usage_scrape/tv_price_refresh/risk_parity_recommend/finnhub_analyst/finnhub_news_insider/trigger_alerts/screen_gov_confluence. sheets.py exposes no upsert/read helper (only 4 write fns, src/sheets.py:10-15).`
- SHEET_ID is hardcoded in the PWA but env-driven on the Python side — config drift waiting to happen. Point Python at a new Sheet and the PWA silently keeps reading the old one. The 50-entry GID table is also hand-maintained with magic numbers and comments admitting some were 'resolved via gspread on a date' — adding/reordering a tab requires manually re-looking-up its GID. — `pwa/src/data.ts:3 (const SHEET_ID = "1N2AAx1...") vs src/sheets.py:128 (os.environ.get("SHEET_ID")); data.ts:14-76 GIDS table with hardcoded numeric gids and notes like 'Live GID resolved via gspread on 2026-05-07' (line 42).`
- schema.py is a 2558-line / 98KB god-module mixing three concerns: 48 tab dataclasses, number/timestamp formatting utilities, and 12 JSON-ingestion mappers (from_ledger/from_grab/from_sidecar). The mappers (1548-1880) are business logic that doesn't belong next to pure schema definitions; editing one tab means scrolling a 2.5k-line file. — `src/schema.py wc -l = 2558; grep shows 48 @dataclass and mapper fns at lines 1548 (snapshot_caspar_from_ledger) through 1783 (decisions_from_ledger).`
- Schema is defined twice with no enforcement: each tab's columns exist as a Python dataclass HEADERS list AND a hand-written TypeScript interface, kept in sync manually. The schema.py docstring even says 'If you change a schema here, update the sheet tab header row to match' — and the PWA interface too. Column drift between the three (Python row, Sheet header, TS interface) is silent until a card renders blank. — `src/schema.py:9 docstring ('If you change a schema here, update the sheet tab header row'); ~40 mirrored interfaces in pwa/src/data.ts (e.g. DecisionRow Python dataclass src/schema.py:110-194 vs TS interface data.ts:344-379). ensure_headers (src/sheets.py:153-155) will silently OVERWRITE row 1 if it drifts, masking mismatches.`
- ensure_headers() is called before every append (multiple times per sync run), each doing a full open_by_key + worksheet + row_values(1) network round-trip just to re-verify headers that almost never change. On the IBKR grab path that's ~6+ redundant header checks per run, padding latency and 429 risk against the same per-minute quota. — `src/sync.py:237-264 (cmd_wsr calls ensure_headers 5+ times in one run); src/sheets.py:139-156 (each ensure_headers does _open_sheet + ss.worksheet + ws.row_values(1)).`
- PWA decision actions are localStorage-only; the documented Sheets write-back never shipped. User-recorded fills/kills/defers don't persist across devices or survive a cache clear, and the brain (which reads decision_queue) can't see what the user actually did. This is a real data-durability gap, though arguably acceptable for a 1-2 user app — flagging the stale TODO, not demanding the enterprise sync. — `pwa/src/lib/decisionActions.ts:6-24 (docstring: 'PWA has read-only access to Sheets'; 'v2 (later): Sync to Sheets via a write endpoint... so actions persist across devices') with persist() writing only to localStorage (lines 87-93). No POST-to-sheet exists anywhere in pwa/src.`
- drive.py carries two complete Drive auth+upload stacks (pydrive2 vs google-api-python-client) because gspread's cached OAuth token format is incompatible with pydrive2. upload_pdf uses pydrive2; upload_file/upload_text use the v3 API. Dead-ish weight and a footgun — a token-format change breaks one path silently while the other works. — `src/drive.py:18-52 (_auth via pydrive2 GoogleAuth) and upload_pdf:55-91; vs src/drive.py:94-130 (_drive_service via googleapiclient) and upload_file:183-236. Comment at line 192-195 explains the duplication ('unlike pydrive2 which needs oauth2client-style creds').`
- Append + audit-timestamp design makes every tab grow unbounded and accumulate duplicates, which the PWA pays for on read (full history every fetch) and which requires a manual cleanup_dupes.py janitor. Some upsert rows self-prune (e.g. macro_alerts_state >7d, schema.py:1141-1142) but the append tabs (positions, options, snapshots, technical_scores) never do. — `scripts/cleanup_dupes.py:1-4 (docstring: 'One-shot cleanup: de-duplicate rows per tab and drop junk rows'); _ts_suffix appends HHMMSS to the date PK so re-runs the same day never collide and always append (src/schema.py:46-48); PWA dedup() runs client-side over full history (data.ts:1370-1377).`
- Staleness is handled ad-hoc and inconsistently per consumer rather than centrally. The PWA re-derives 'latest' client-side for most tabs, but special-cases freshness for a few (harvest_scan cross-date guard; live_prices freshness chip), and the gviz read has no last-updated signal — so a frozen cron silently serves yesterday's 'latest' with no staleness flag on most cards. — `pwa/src/data.ts:1532-1537 (bespoke harvestScan staleness guard comparing dates to scan_results); data.ts:1454/1510 (livePricesUpdatedAt freshness only for one tab); latest()/latestGroup() (data.ts:1354-1363) blindly take max-date with no age check for the other ~35 tabs.`

### Backend scan / scoring engine — the systematic options + growth pipeline that fetches market data (yfinance), computes 16 technical signals, scores them per-strategy ([-100,+100]), finds optimal-delta option strikes, ranks candidates into a 0-100 composite, and assembles the daily_plan that the Alpaca paper-executor trades verbatim. Two scheduled engines run independently: daily_tracker.py (portfolio + watchlist CSP/CC scan) at its own cron, and daily_options_scan.py (unified watchlist all-strategies + FinViz high-IV discovery) at 10:35 SGT. build_daily_plan.py aggregates their sheet output at 14:00 UTC. All state lives in Google Sheets tabs; no DB.
