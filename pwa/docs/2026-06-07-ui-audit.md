# FinancePWA — UI/UX Audit (2026-06-07)

Scope: `pwa/` only. 11 tab pages, ~41 cards, a new `src/components/ui/` primitive
library, and a mature CSS motion/token system in `src/index.css`. The app is a
live, phone-first dark "fintech terminal" (Bloomberg-amber accent, glass cards,
CRT scanline). The aesthetic is already strong and coherent — this audit looks
for *polish* gaps, not a redesign.

Method: read `App.tsx`, `index.css` (tokens + motion utilities), the `ui/`
primitives, `AsyncStates`, `HomePage`, `DecisionsPage`, `DecisionCard`,
`TabBar`, `StickyTabs`, `PullToRefresh`, `StockDetail`, `DailyBriefCard`, plus
grep surveys for `focus-visible`, `role="button"`, `onClick={onOpen}`, and tap
targets. Baseline `npm run build` passes; `npm run lint` reports 7 errors
(5 pre-existing app bugs out of scope, 2 are the StatusPill nit, below).

---

## Verdict

The visual layer (surfaces, color, type tokens, atmospheric background) is
genuinely high-end. The gaps are in **motion continuity** (the app animates
*content within* a page but never the **page swap itself** — tab changes are a
hard cut), **accessibility** (`focus-visible` exists in only 3 of ~30
interactive files; the entire top nav, app header, and modal chrome are
keyboard-invisible), and a handful of **consistency/branding** wins (logo absent
from the running app; one-off chip/pill markup that the new `ui/` primitives
already cover). None of these are broken — they are the difference between "very
good" and "production-grade polished."

---

## Ranked findings

### P0 — High impact, safe, in-scope

**1. Page swaps are a hard cut — no transition on tab change.**
`App.tsx:112-218` `renderPage()` switches on `tab` and the new page mounts with
zero entrance motion. Cards *inside* a page stagger in via `fade-up` (e.g.
`HomePage.tsx:92,126,150…`), but the page-level container doesn't, so switching
tabs feels instant/flat versus the polished in-page reveal. The scroll-to-top on
tab change (`App.tsx:83-85`) compounds the abruptness.
→ Fix: wrap the page render in a keyed container that re-triggers a lightweight
entrance keyframe on `tab` change. Pure CSS, gated by reduced-motion.

**2. `focus-visible` is almost entirely absent — keyboard/switch users are lost.**
Only `AsyncStates.tsx`, `ui/ActionButton.tsx`, `SettingsPage.tsx` define focus
rings (grep: 3 files). Missing on every primary surface:
- App header Search + Refresh buttons — `App.tsx:260-281` (`active:scale-90` only).
- TabBar — all 8 nav buttons, `TabBar.tsx:62-68` (no focus style at all).
- StickyTabs — segmented control buttons, `StickyTabs.tsx:49-52`.
- StockDetail — Back + close (X) buttons, `StockDetail.tsx:404-411, 421-423`.
- DecisionCard — the `role="button"` card has `tabIndex={0}` + key handler
  (`DecisionCard.tsx:494-499`) but no visible focus ring.
→ Fix: add a shared `.focusable` utility (one focus-visible ring recipe) in
`index.css` and apply to these high-traffic controls. WCAG 2.4.7.

**3. The app logo never appears in the running app.**
`public/icon-192.png` is used only on the sign-in gate (`AsyncStates.tsx:223`).
The in-app header is text-only — `App.tsx:246-257` ("title + 'Casaa Finance'").
A small logo mark in the header is the single highest-ROI branding touch and was
explicitly requested.
→ Fix: add a ~22px rounded logo to the left of the header title.

**4. StatusPill exports non-components → react-refresh eslint error (known nit).**
`ui/StatusPill.tsx:48` (`DECISION_STATUS`) and `:57` (`resolveStatus`) trip
`react-refresh/only-export-components` (confirmed via `npm run lint`). These are
consumed by `DecisionsPage.tsx:38` and `DecisionCard.tsx:29`.
→ Fix: move the const + helper into a sibling `statusConfig.ts` and re-export
through `ui/index.ts`, leaving `StatusPill.tsx` component-only. Zero behavior
change.

### P1 — Good polish, low risk

**5. Card press has no micro-interaction.**
`.glass` has a hover transform transition (`index.css:205`) but the comment at
`index.css:291-297` documents that a global `:active` was *deliberately removed*
(it caused hit-area shifts). Result: tappable cards (DailyBriefCard,
WsrSummaryCard, PaperStatusCard, DecisionCard) have only `active:opacity-*` or
`active:bg-white/3` — no tactile press scale. A scoped, opt-in press utility
(applied to the inner element, never the glass container) restores tactility
without the documented bug.
→ Fix: add a `.press` utility (`active:scale-[0.985]`) and apply to the
DecisionCard root + the header icon buttons. Keep it off `.glass` itself.

**6. Header icon buttons are 36px — below the 44px tap-target guideline.**
`App.tsx:262,271` render `w-9 h-9` (36px) Search/Refresh buttons. Apple HIG and
WCAG 2.5.5 want ≥44px. The visual box can stay 36px; the *hit area* should be
≥44px.
→ Fix: keep the 36px visual chip, add `min-w-11 min-h-11`-equivalent hit padding
(or bump to `w-10 h-10` + the focusable ring). Low risk.

**7. Tab-bar buttons lack an explicit min tap height.**
`TabBar.tsx:62-68` buttons are `pt-2 pb-1.5` inside a 72px bar — vertically fine
in practice, but there's no guaranteed min and no focus affordance (see #2).
→ Fix: covered by the focusable utility + the bar's existing height; add
`focus-visible` only.

**8. `transition-all` used on animated controls (minor perf / repaint).**
`App.tsx:262,271`, `ActionButton.tsx:69`, StickyTabs inline. `transition-all`
animates every property (incl. layout) — best practice is to transition only
`transform`/`opacity`/`color`/`box-shadow`. Cosmetic, but a clean win where
touched.
→ Fix: narrow to specific properties on the surfaces being edited (don't sweep
all 41 cards).

**9. Reduced-motion list is incomplete.**
`index.css:415-419` strips `fade-up{,-1..-4}` but the `.fade-up-5` delay class
(`index.css:379`), `.slide-in`, `.scale-in` (already listed), and the new
page-transition + spin won't all be covered as motion grows. The wildcard at
`:420-423` clamps transitions to ~instant, which mostly saves it, but the
animation list should explicitly include any new keyframes.
→ Fix: add the new page-transition + `.press`/`.tab-pop` names to the explicit
reduced-motion block.

### P2 — Consistency / nice-to-have (mostly deferred)

**10. One-off pill markup where `ui/` primitives now exist.**
`DecisionsPage.tsx:256-268` hand-rolls status-count pills from `DECISION_STATUS`
instead of a primitive; `DecisionCard.tsx:557-560` hand-rolls a "bucket" chip
that `Chip` covers. These are visually correct (they read the same token map) so
the risk/reward of churning them is low — flagged for awareness, not all fixed,
per the "don't rewrite 41 cards" constraint.

**11. Inline-styled sub-panels in DecisionCard.**
`DecisionCard.tsx:67-101` (AccumulationPlanRow) and `:119-176` (OptionsSpecRow)
use raw inline `style={{…}}` blocks rather than tokens/utilities. Functional and
isolated; deferred.

**12. `--t-tiny` (9px) / 8px literals for timestamps.**
`App.tsx:251,253` use `text-[8px]` for build + refresh time — below the 9px
`--t-tiny` token and quite small. It's deliberately de-emphasized chrome, but
8px is borderline. Minor; left as-is (intentional density).

**13. Loading/empty/error coverage is good.**
`AsyncStates` (Loading/Empty/Error/NotAuthorized/SignIn) is well-built with
`role`/`aria-live`/`aria-busy`. Cards mostly have their own skeletons
(DailyBriefCard, etc.) and null-render on empty. No gap worth a code change;
noted as a strength.

## What I'm NOT touching (and why)

- The 5 non-StatusPill eslint errors (`PnlCard`/`TldrTodayCard` `Date.now`
  purity, `HarvestPage` conditional hooks, `ActiveHarvestCard` export,
  `settings.ts` deps) — pre-existing logic issues, out of an *UI polish* scope,
  and risky to "fix" blindly on a live app.
- Data joins, fetch/refresh logic, every page's behavior — untouched by design.
- Mass card migration to primitives (#10/#11) — explicitly out of scope.
- The dark theme, color system, glass surfaces, scanline — preserved exactly.
